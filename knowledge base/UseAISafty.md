### 第三层：Git 守门员 —— Aider (CLI 终端)

## 3. 安全与合规性配置（核心保障）

为了确保代码资产绝对不用于模型训练且不外泄：

1. **全局 Ignore 策略** ：

* 在项目根目录设置 `.gitignore`。
* 在工具端设置专属忽略文件（如 `.cursorignore`, `.aider.conf.yml`）。
* **严禁** Agent 访问 `.env`、秘钥目录及非代码资产文件夹。

1. **API 端零保留协议** ：

* 在订阅平台（如 AiZolo）中勾选 **Zero Data Retention (ZDR)** 选项。
* 优先接入部署在 **Azure OpenAI** 或 **AWS Bedrock** 上的模型节点，这些节点受法律协议约束，绝对禁止数据用于训练。

1. **内网暴露防护** ：

* 由于 Agent 会执行终端命令，必须在 Cline/Aider 中限制其“网络访问”权限（只读或禁用），防止 Agent 自行将代码推送到公网。

### 一、 核心工具详解：什么是 Cline、RooCode 和 Aider？

这三个工具是目前 AI 编程领域“Agent 模式”的代表：

1. **Cline (原名 Claude Dev)：**
   * 它是一个 VS Code 插件，但它不只是补全代码。它是一个**拥有文件读写权限和终端执行权限**的智能体。
   * **特点：** 它会先“思考”并列出计划，在得到你许可后，它能自主创建文件、修改现有代码、运行编译器报错、根据报错再修改，直到任务完成。它支持 MCP（Model Context Protocol）协议，可以调用外部工具。
2. **RooCode：**
   * 它是 Cline 的一个深度定制/分支版本，在 2026 年非常流行。
   * **特点：** 它强化了“模式切换”。你可以把它切换到 **Architect Mode（架构师模式）**来设计游戏引擎模块，或 **Code Mode（编码模式）**来写业务 C# 代码。它对本地代码库的索引（Indexing）做得比 Cline 更深，能更精准地理解跨文件的复杂逻辑。
3. **Aider：**
   * 这是一个**基于 CLI（命令行）**的 AI 配对编程工具。
   * **特点：** 它与 Git 深度绑定。当你让它重构一个 Unity 脚本时，它会自动理解你的 Git 树，修改完后会自动提交（Commit）并附带详细的说明。它在处理“严密的逻辑推理”时表现极佳，因为它强迫 AI 在一个闭环的“编辑-测试-修复”循环中工作。

* **Aider：** 目前公认的 CLI 编程天花板。它通过终端直接操作 Git 仓库。它的逻辑推理表现极其严密，因为它强迫模型进入“分析 -> 修改 -> 提交 -> 测试 -> 修复”的闭环。

* **OpenRouter / 硅基流动 (SiliconFlow)：** 这些是“API 转发超市”。你只需要买一个平台的额度，就能调用全世界（GPT, Claude, Llama, DeepSeek, MiniMax）的所有模型。对于想要“随换随用”的你，这是必不可少的后端。

## 1. 为什么“网页贴代码”是低效的“人工沙盒”？

你提到的方案 A（网页对话）确实能解决安全问题，但在处理 **2022.3.20f1** 这种大型 Unity 项目时，它会遇到三个致命瓶颈：

* **依赖地狱（Context Loss）：** 你贴一段代码给 Gemini，它能改好，但它不知道这段代码在 `ProjectSettings` 里的配置，也不知道它被哪个 `Asmdef` 引用。**CLI Agent（如 Aider/OpenCode）** 的核心竞争力在于  **Codebase Map（代码地图）** 。它会扫描整个项目的符号表，即使你没给它读某个文件，它也知道那个文件的接口定义。
* **多文件协同（Atomicity）：** 网页对话一次只能给你一段文本。Agent 可以在一次对话中 **跨文件修改** 。比如你改了一个底层 C++ 引擎模块的导出接口，它能同时帮你更新 5 个 C# 脚本的调用逻辑。
* **反馈闭环（The Loop）：** 在网页里，你贴代码 -> AI 给代码 -> 你粘贴 -> 运行报错 -> 你再拷报错贴回去。CLI Agent 可以直接运行 `dotnet build` 或自定义的 `Batch` 脚本，看到报错立刻 **自发进行第二轮迭代** ，直到编译通过。

**结论：** 网页对话是“静态咨询”，CLI Agent 是“动态协作”。

## 2. 深度阅读 OpenCode 源码：这才是你的“黄金路线”

我非常支持你 **“不 Fork，只阅读与拦截”** 的想法。这不仅是出于安全，更是因为在 2026 年， **理解 AI Agent 的架构本身就是极高价值的技术资产** 。

如果你去读 OpenCode 或 Aider 的源码，你会发现它们的核心由以下三个模块组成，这也是你最需要“审计”的地方：

### A. 上下文构造引擎 (Context Assembly)

* **审计重点：** 它是如何通过 `.gitignore` 过滤文件的？它是否在偷偷扫描你的 `Library` 或 `Temp` 文件夹？
* **学习点：** 学习如何利用 **AST（抽象语法树）** 对代码进行摘要，而不是全量上传。这能帮你省下巨额的 Token 费用。

* **生态封闭性** ：JetBrains 极力推销自家的  **AI Assistant** ，这导致它对第三方插件的支持（尤其是像 Continue 这种需要深度控制 IDE 内部文件的插件）响应非常缓慢。

* **Codeium 的误区** ：在 Rider 插件商店中，你应该搜索 **“Codeium”** 而不是 “Codeium Enterprise”。普通版 Codeium 是有 Rider 支持的，但它依然要求登录账户且闭源， **不推荐给你** 。

## 一、 AI Agent API 数据包结构拆解

无论是 Aider 还是 OpenCode，它们在与云端大模型（如 MiniMax、Claude、GPT-5）通信时，本质上都遵循 **OpenAI 兼容的 Chat Completions API** 格式。

一个典型的 POST 请求数据包结构如下（以 JSON 为例）：

### 1. HTTP Header (请求头)

这是防火墙审计的第一道防线。

**HTTP**

```
POST /v1/chat/completions HTTP/1.1
Host: api.minimax.chat (或你的本地代理地址)
Authorization: Bearer sk-xxxxxxxxxxxx  // 你的 API Key
Content-Type: application/json
```

### 2. Payload (请求体)

这是 AI Agent 将你的代码发送到云端的核心部分，也是最敏感的部分。

**JSON**

```
{
  "model": "gpt-5-low-latency", // 或 minimax-coding-special
  "messages": [
    {
      "role": "developer", // 2025年后普及的系统角色，优先级最高
      "content": "你是一个 Unity 6 专家。你只能根据项目地图修改代码。"
    },
    {
      "role": "user",
      "content": "基于当前 Repo Map，重构 PlayerController.cs 以支持双段跳。"
    },
    {
      "role": "assistant", // 包含之前的对话历史，确保 Agent 具备记忆
      "content": "我已阅读当前脚本，正在准备修改方案..."
    }
  ],
  "tools": [ // Agent 的“手”，让它可以读写文件
    {
      "type": "function",
      "function": {
        "name": "read_file",
        "parameters": { "path": "Assets/Scripts/PlayerController.cs" }
      }
    }
  ],
  "reasoning_effort": "high", // 2026年推理模型特有的深度思考参数
  "stream": true // 流式输出，让你实时看到代码一行行蹦出来
}
```

**你的防火墙审计点：** 重点扫描 `messages` 数组中的 `content` 字段。如果里面包含了 `.env` 文件的内容或非代码类的敏感资产路径，你的本地代理应立即阻断请求。

## 二、 UnrealClaude 深度拆解：MCP 协议如何让 AI “看见”虚幻引擎

`UnrealClaude`（以及类似的 `UnrealMCP` 插件）的核心逻辑是利用了 Anthropic 推出的 **MCP (Model Context Protocol)** 协议。它像一个“USB-C 接口”，将 Unreal Engine 的内部数据标准化后喂给 AI。

### 1. 架构图解

**Plaintext**

```
[Claude Code CLI] <---(JSON-RPC over stdio)---> [MCP Server (Node/Python)] <---(Remote Control API/C++)---> [Unreal Engine Editor]
```

### 2. 它是如何“看见”资产的？

在 MCP 架构中，资产并非“全量上传”，而是通过 **“工具调用（Tool Calling）”** 实现按需查询。

* **暴露工具** ：MCP Server 向 AI 声明自己拥有一系列工具，如 `list_assets`、`get_actor_properties`、`query_blueprint`。
* **AI 发起查询** ：当你问“场景里有哪些灯光？”时，Claude 会向 MCP Server 发送一个请求：`call_tool: list_actors(class="PointLight")`。

## 1. 关于“暗文与加密”的真相

你担心如果数据被加密了无法过滤，这里的核心在于理解 **加密发生的层次** ：

* **传输层加密 (TLS/HTTPS)** ：目前所有的工具（Aider, OpenCode, Claude CLI）默认都是通过 HTTPS 发送到服务器的。这意味着在网络电缆上，你确实只能看到加密后的流量。
* **如何“破局”** ：
* **本地拦截 (Application Layer)** ：因为你是工具的使用者，你可以通过设置环境变量（如 `HTTPS_PROXY`）将流量引导至你编写的 **本地 C# 审计代理** 。在这个代理中，你使用自签名的证书进行“中间人拦截（MITM）”，此时数据包在进入正式加密传输前，对你来说是 **完全透明的 JSON 明文** 。
* **源码插桩** ：既然你要读 Aider（Python）或 OpenCode（TypeScript）的源码，最极客的做法是直接在它们调用 `requests` 或 `fetch` 的函数处加一行代码，将 payload 直接打印到本地日志。

> **补充：** 只要 AI 模型需要“理解”你的代码，它就必须接收 **明文（或分词后的 Token）** ，模型厂商目前不会在客户端进行无法解密的私有协议加密，因为那会破坏 API 的通用性。

## 2. 关于项目资产索引：并不是“全量拷贝”

* **按需读取 (Lazy Loading)** ：只有当 AI 明确发出 `read_file` 的指令时，Agent 才会读取具体文件内容并放入 JSON 发送。
* **安全风险点** ：你需要过滤的不仅是“代码内容”，更重要的是**“元数据”**。有些 Agent 会尝试读取你的 `.git/config` 或全局环境变量，这些数据块很小，极易在你不经意间混入 JSON 传走。

### 四、 解决“Inspector 盲区”：给你的第一把利器

为了兑现之前的承诺，并配合你**“局部重构、严格控制”**的工作流，我为你写好了这个 Unity Editor 脚本。它能直接解决 AI 看不见 Inspector 数据的问题。

**功能：** 在 Unity 菜单栏增加 `Tools/AI/Copy Inspector Context`，一键提取选定脚本的变量值并生成 AI 可读的快照。

**C#**

```
using UnityEngine;
using UnityEditor;
using System.Text;
using System.Reflection;

public class AIInspectorContextHelper : EditorWindow
{
    [MenuItem("Tools/AI/Copy Inspector Context")]
    public static void CopyContext()
    {
        GameObject target = Selection.activeGameObject;
        if (target == null) return;

        StringBuilder sb = new StringBuilder();
        sb.AppendLine("// [AI_CONTEXT_START]");
        sb.AppendLine($"// Target Object: {target.name}");

        MonoBehaviour[] scripts = target.GetComponents<MonoBehaviour>();
        foreach (var script in scripts)
        {
            if (script == null) continue;
            sb.AppendLine($"// --- Script: {script.GetType().Name} ---");
          
            FieldInfo[] fields = script.GetType().GetFields(BindingFlags.Public | BindingFlags.NonPublic | BindingFlags.Instance);
            foreach (var field in fields)
            {
                // 只抓取 SerializeField 或 Public 变量
                if (field.IsPublic || System.Attribute.IsDefined(field, typeof(SerializeField)))
                {
                    object val = field.GetValue(script);
                    sb.AppendLine($"// {field.FieldType.Name} {field.Name} = {val ?? "null"};");
                }
            }
        }
        sb.AppendLine("// [AI_CONTEXT_END]");

        EditorGUIUtility.systemCopyBuffer = sb.ToString();
        Debug.Log("AI Context copied to clipboard!");
    }
}

## 一、 2026 年“龙虾系”Agent 助手安全隐患汇总

这些所谓的“一键部署”助手，其核心逻辑通常是基于 **VNC/控制流** 或  **屏幕视觉解析** ，这导致了它们与传统 IDE 插件完全不同的安全漏洞。

### 1. ArkClaw & QClaw：间接提示词注入（Indirect Prompt Injection）

* **漏洞描述** ：这是 2026 年初最猖獗的攻击方式。攻击者在网页、PDF 甚至是代码注释中隐藏了一段“隐形指令”。当 ArkClaw 帮你查资料或读代码时，这段指令会接管 Agent 的控制权。
* **后果** ：Agent 可能会被诱导执行 `rm -rf /` 或将你的 `.env` 配置文件通过 HTTP 请求发送到黑客服务器。
* **风险等级** ： **致命** 。

### 2. KimiClaw & WorkBuddy：屏幕截图泄露（Screen-Scraping Leak）

* **漏洞描述** ：这些助手为了理解上下文，会频繁截图上传到云端进行多模态分析。
* **漏洞信息** ：2026 年 2 月，某知名论坛曝出 KimiClaw 的云端缓存未加密，导致部分用户的银行流水、私密聊天记录截图在暗网流出。
* **风险点** ：它们不只是看代码，它们“看”到了你屏幕上的一切。

### 3. OpenClaw & JVSClaw：会话劫持与 Token 持久化

* **漏洞描述** ：这些开源项目为了方便，往往将各种 API Key（包括你的 GitHub Token、AWS Key）明文保存在本地 `config` 文件夹中。
* **安全报告** ：近期发现一种针对 OpenClaw 的恶意脚本，专门扫描此类路径并一键打包上传。

## 为什么我决定更新清单？（安全漏洞复盘）

### 1. 权限越界：OS Agent vs Dev Agent

“龙虾系”助手（如 OpenClaw）追求的是接管你的电脑，这导致了 **“混淆代理（Confused Deputy）”** 问题。

* **风险** ：如果你在看一个带毒的技术文档（或网页），黑客可以利用网页里的隐藏指令，诱导你的 OpenClaw 去读取你的物理机 SSH Key 并发给外部服务器。
* **建议** ：只使用限制在 **“项目文件夹”** 视角的开发工具（Aider/Roo Code），永远不要给 AI 操作桌面的权限。

### 2. 内存毒化与持久化后门

最近爆出的 `SOUL.md` 和 `HEARTBEAT.md` 漏洞，揭示了 Agent 的持久化记忆可以被“毒化”。

* **漏洞** ：一旦 AI 被注入了恶意指令，这些指令会像“思想钢印”一样刻在它的记忆文件里，即便你重启工具，它依然会潜伏并在你下次部署项目时插入后门代码。
* **更新策略** ：**严禁**让 AI 助手管理其自身的系统提示词（System Prompt）或记忆文件。你必须亲自审计这些文件。

### 3. MCP（Model Context Protocol）的滥用

OpenCode 等工具支持 MCP，这本来是好事。但在 2026 年，恶意“Skill/工具包”已经在社区流传。

* **建议** ：不要安装任何未经审核的第三方 MCP 工具。只使用你亲自阅读过源码的本地 MCP Server（例如连接 Unity 6 的那个）。

### 一、 C# 版：极简 API 流量审计网关

这个脚本基于 .NET 8.0 构建。它作为一个本地中转站（Proxy），拦截并解密从 Continue 或 Roo Code 发往 MiniMax 等云端的一切数据。

**C#**

```
using System;
using System.IO;
using System.IO.Compression;
using System.Net;
using System.Text;
using System.Threading.Tasks;
using System.Collections.Generic;

// 这是一个本地审计网关，原理是将 AI 工具的 API 地址指向这里 (localhost:8080)
class DevFirewall
{
    private static readonly string LogPath = $"Audit_Report_{DateTime.Now:yyyyMMdd}.txt";
    private static readonly List<string> SensitiveKeywords = new() { "Password", "Secret_Key", "Assets/Private", ".env" };

    static async Task Main()
    {
        var listener = new HttpListener();
        listener.Prefixes.Add("http://127.0.0.1:8080/");
        listener.Start();
        Console.WriteLine("🛡️ 2026 开发防火墙已启动，监听端口: 8080...");

        while (true)
        {
            var context = await listener.GetContextAsync();
            _ = Task.Run(() => ProcessRequest(context));
        }
    }

    private static async Task ProcessRequest(HttpListenerContext context)
    {
        try
        {
            var request = context.Request;
          
            // 1. 处理自动解压 (AI 工具为了节省流量，通常发送 gzip 压缩包)
            Stream bodyStream = request.InputStream;
            if (request.Headers["Content-Encoding"] == "gzip")
            {
                bodyStream = new GZipStream(request.InputStream, CompressionMode.Decompress);
            }

            using var reader = new StreamReader(bodyStream, Encoding.UTF8);
            string rawJson = await reader.ReadToEndAsync();

            // 2. 深度审计：观察 AI 发送了什么数据包结构
            /*
               【API 数据包内部结构拆解】：
               - "model": 指定使用的模型 (如 minimax-coding-01)
               - "messages": 包含对话上下文。这是重灾区，里面有 AI 抓取的代码片段。
               - "tools": 【核心】如果启用了 MCP，这里会列出 AI 拥有的权限（如 read_file, run_command）。
               - "tool_calls": AI 决定执行某个指令时发出的具体参数。
            */
          
            Console.WriteLine($"\n[流量审计] 来自: {request.UserHostAddress} | 长度: {rawJson.Length} bytes");
          
            // 3. 全文搜索敏感词
            bool hasThreat = false;
            foreach (var key in SensitiveKeywords)
            {
                if (rawJson.Contains(key, StringComparison.OrdinalIgnoreCase))
                {
                    Console.ForegroundColor = ConsoleColor.Red;
                    Console.WriteLine($"⚠️ 拦截到潜在泄露！发现关键字: {key}");
                    Console.ResetColor();
                    hasThreat = true;
                }
            }

            // 4. 生成每日审计报告
            await File.AppendAllTextAsync(LogPath, $"[{DateTime.Now:HH:mm:ss}] {rawJson}\n---\n");

            // 5. 转发给真实的 API 服务器 (此处仅演示拦截逻辑)
            // 在实际生产中，你可以使用 HttpClient 将 rawJson 转发给 api.minimax.chat
          
            context.Response.StatusCode = hasThreat ? 403 : 200;
            context.Response.Close();
        }
        catch (Exception ex) { Console.WriteLine($"❌ 审计错误: {ex.Message}"); }
    }
}
```

### 二、 Python 版（等价逻辑）

如果你希望在轻量级终端环境中运行，可以使用 Python 版：

**Python**

```
import gzip
from flask import Flask, request, Response
import datetime

app = Flask(__name__)
LOG_FILE = f"Audit_Report_{datetime.date.today()}.log"
SENSITIVE_WORDS = ["Password", "Secret_Key", ".env", "Assets/Private"]

@app.route('/', defaults={'path': ''}, methods=['POST'])
@app.route('/<path:path>', methods=['POST'])
def proxy(path):
    # 1. 自动解压 Gzip
    raw_data = request.get_data()
    if request.headers.get('Content-Encoding') == 'gzip':
        content = gzip.decompress(raw_data).decode('utf-8')
    else:
        content = raw_data.decode('utf-8')

    # 2. 拆解分析：AI 到底发了什么？
    # 观察内容里的 "messages" 节点，你会发现 AI 为了回答你的问题，
    # 会把你的 Unity 脚本（甚至是整个项目结构）作为 Context 塞进去。
  
    # 3. 敏感词检索与拦截
    for word in SENSITIVE_WORDS:
        if word in content:
            print(f"🔥 严重警告：AI 正在尝试发送敏感信息 [{word}]")
            return Response("Blocked by Local Firewall", status=403)

    # 4. 写入审计日志
    with open(LOG_FILE, "a", encoding="utf-8") as f:
        f.write(f"[{datetime.datetime.now()}] {content}\n\n")

    print(f"✅ 已记录数据包 ({len(content)} 字符)")
    return Response("OK", status=200)

if __name__ == '__main__':
    app.run(port=8080)

### 三、 逐步拆解：AI 到底往云端发了什么？

当你使用 **Roo Code** 时，你的 API 包通常包含以下三个危险层：

1. **Repo Map（项目指纹）** ：即使你不发文件，Agent 也会先扫描你的目录树。它会把你的文件名、文件夹结构发出去。如果你的文件夹命名包含敏感项目代号（如 `Project_Secret_RPG`），这一步就已经泄露了。
2. **Context Injection（上下文注入）** ：为了让你修复 Bug，Roo Code 会读取你当前选中的脚本。你会发现 `messages` 数组里有一个 `role: user` 的消息，后面跟着数千行的源代码。
3. **MCP Tool Definition（权限申明）** ：AI 会把“我能干什么”发给云端，让云端决定。例如：`"tools": [{"name": "read_file"}]`。

### 四、 如何拦截并观察 MCP 指令？

你截图中显示的 **MCP 服务器商城（Marketplace）** 是获取插件的地方。但 **MCP 的实际工作原理** 是：

* **Roo Code** （客户端）启动了本地的  **MCP Server** （例如一个 Unity 桥接脚本）。
* 当 AI 决定要读取一个资产时，它会在发往云端的 JSON 中加入 `tool_calls` 节点，里面写着：`{"call": "unity_get_asset", "params": {"name": "Hero.prefab"}}`。
* **如何通过网关拦截？**
  所有的 MCP 指令在变成物理操作前，都必须经过云端模型的“批准”。因此，**只要你观察发往 API 的 `tool_calls` 或云端返回的 `choices` 节点，你就能预判 AI 下一秒要对你的项目做什么。**

### 五、 VS Code MCP 工作提示
